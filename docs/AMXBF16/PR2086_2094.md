## 1. 结论摘要

PR #2094 是一个建立在 PR #2086 当前 head 上的 stacked PR。相对本地 `1e95053`，远端 head 没有分叉，而是线性领先 3 个提交，因此当前可以 fast-forward；但这 3 个提交不都属于 #2094 自身的新设计：

```text
本地 HEAD
1e95053  [docs](sft): remove Qwen3 Full-FT performance report
   │
   ├─ 66f5f15  [perf](kt-kernel): reduce BF16 Full-FT checkpoint overhead
   │            PR #2086 相对本地新增的 checkpoint/cache/dWeight 优化
   │
   ├─ 6eeffa7  [perf](kt-kernel): make SFT optimizer gradients authoritative
   │            PR #2094 的主体实现
   │
   └─ e26e21f  [perf](kt-kernel): avoid eager Full-FT gradient zeroing
                PR #2094 的补充 first-touch 优化，也是当前 PR head
```

整体作用可概括为四点：

1. **保留 gradient checkpoint，但不再重复计算 CPU MoE forward。** 第一次 forward 就保留 C++ expert cache；checkpoint recompute 仍重跑 attention、norm 和 router，但复用第一次的 CPU expert 输出和 cache。
2. **C++ BF16 gradient buffer 成为唯一的 optimizer gradient。** `Parameter.grad` 直接别名到 C++ buffer，Autograd 不再返回三份巨型 base gradient 给 `AccumulateGrad`，从而去掉第二份约 54 GiB Full-FT 梯度、首个 microbatch 的大复制以及后续 GAS microbatch 的大范围 `aten::add_`。
3. **把梯度窗口语义下沉到 C++。** optimizer window 的第一个 microbatch 覆盖写，后续 microbatch 原地累积；新窗口只清理可能残留的 expert slice，而不是每步扫描全部 optimizer-gradient buffer。
4. **优化 BF16 dWeight 数据准备。** input 和 route-weighted `dY` panel 按 expert 预打包一次并被多个 `I` tile 复用，gradient store 同时支持覆盖、累积和缩放。

这组修改与此前“关闭 checkpoint + 消除全量 buffer clear”的设想不同：它没有关闭 PyTorch checkpoint，也没有删除所有清零。它只跳过 checkpoint 重算中的 **CPU MoE 部分**，并消除/缩小 **持久 optimizer-gradient buffer** 的稳定阶段清零；attention/router 重算、CPU 输出回传 GPU、每次 backward 的临时 partial-buffer 清零仍然存在。

## 3. `66f5f15`：保留 checkpoint，但复用第一次 CPU MoE forward

提交：[66f5f15](https://github.com/kvcache-ai/ktransformers/commit/66f5f15a7947f5ad03a71eec514c408148d4cc77)

### 3.1 修改前

本地 `1e95053` 在 non-reentrant checkpoint 下的单层流程是：

```text
第一次 forward
  GPU attention/norm/router
  CPU MoE forward(save_for_backward=false)
  返回输出，但不保留 C++ backward cache

backward 到达该层
  ctx.saved_tensors 触发整个 decoder layer recompute
  GPU attention/norm/router 再执行
  hidden/id/weight 再从 GPU staging 到 CPU
  CPU MoE forward(save_for_backward=true) 再执行并建立 cache
  CPU MoE backward 消费 cache
```

这样可以让 48 层共享少量 CPU cache，但每步多执行一次真实 CPU expert forward。

### 3.2 修改后

```text
第一次 forward
  GPU attention/norm/router
  CPU MoE forward(save_for_backward=true)
  C++ cache 保留到该层 backward
  CPU expert output 另存一份供 checkpoint recompute 使用

backward 到达该层
  ctx.saved_tensors 仍触发 decoder layer recompute
  GPU attention/norm/router 仍再执行
  不再提交 hidden/id/weight 到 CPU，也不再执行 CPU MoE forward
  从 CPU 缓存取 expert output 并传回 GPU，使 recompute 图继续成立
  CPU MoE backward 消费第一次 forward 留下的 cache
```

也就是说：

- `kt.sft.checkpoint_recompute` 不会消失；
- 被跳过的是重算中的 CPU input staging、CPU MoE kernel 和对应同步；
- cached expert output 的 CPU→GPU 传输仍在；
- 第一次 forward 现在要保存 cache 和一份 CPU output，部分工作从 backward 计时移动到了 forward；
- backward/forward 比值可能因“少算 backward、增加 forward 保存流量”而改善，必须同时看完整 step/TPS，不能只看比值。

### 3.3 启用范围

该优化默认由 `KT_REUSE_CHECKPOINT_FORWARD` 控制，默认值为开启，但同时要求：

```text
single process
full_weight_grad = true
lora_rank = 0             # pure Full，不是 Hybrid
method = AMXBF16_SFT
KT_REUSE_CHECKPOINT_FORWARD != 0
```

多进程、LoRA、Hybrid、INT8/INT4、SkipLoRA 和 GPU routed experts 不使用这条 checkpoint-reuse 路径。设置 `KT_REUSE_CHECKPOINT_FORWARD=0` 可回到旧的 CPU MoE recompute 行为。

### 3.4 为什么必须关闭跨层 shared cache

第一次 forward 留下的 C++ activation 要一直保存到逆序 backward，因此 48 层不能继续共用同一 cache pool。代码在启用复用时强制：

```text
wrapper.share_cache_pool = false
```

按当前 Qwen3-30B-A3B 配置 `T=4096, H=2048, top-k=8, I_local=384, 2 NUMA`，一个 cache slot 是：

```text
input          = 4096 × 2048 × BF16                  =  16 MiB / NUMA
gate/up/inter  = 3 × 4096 × 8 × 384 × BF16          =  72 MiB / NUMA
down output    = 4096 × 8 × 2048 × BF16             = 128 MiB / NUMA
合计                                                   216 MiB / NUMA / slot
两 NUMA                                                432 MiB / layer / slot
```

默认 `kt_max_cache_depth=2` 时：

```text
本地 shared pool             = 432 MiB × 2             = 0.844 GiB
PR 每层独立 cache            = 432 MiB × 2 × 48        = 40.50 GiB
cache 净增                                                39.656 GiB
48 层 cached CPU output      = 4096 × 2048 × 2 × 48    = 0.750 GiB
checkpoint-reuse host 增量                               ≈ 40.406 GiB
```

当前每层只有一份未消费 cache 的常规训练流程理论上可使用 `kt_max_cache_depth=1`，使独立 cache 降为 20.25 GiB；但这是配置优化建议，不是 PR 自动完成的变化，必须验证 cache stack、异常恢复和短训后再采用。

这部分增加的是 CPU DRAM，不是 GPU activation 显存。由于 PyTorch gradient checkpoint 本身仍开启，GPU 不会承担完整 no-checkpoint activation 集合。

### 3.5 同一提交的其他优化

- Full-FT BF16 forward 的部分输出直接写入 backward cache，减少“先写计算 buffer、再复制到 cache”的流量。
- BF16 dWeight 先按 expert 预打包共享 input panel 和 route-weighted `dY` panel，再让各个 `I` tile 重用，减少重复动态 packing。
- AVX512 fallback 改为 register-blocked 分块，降低通用回退路径的 accumulator spill。
- base gradient 首次/指针变化时仍完整初始化；稳定新窗口中，当前 active expert 由 dWeight 完整覆盖，只清“上一窗口 active、当前窗口 inactive”的 stale expert。
- profiler 新增 cache 元数据/输入、panel pack、gradient store、clear bytes 和 checkpoint cached-CPU-MoE 等阶段。

## 4. `6eeffa7`：C++ buffer 成为唯一的 `Parameter.grad`

提交：[6eeffa7](https://github.com/kvcache-ai/ktransformers/commit/6eeffa77a45fa3cbf5c3e9e6e5b2a783627b3c3f)

### 4.1 本地旧路径

```text
C++ 计算 dW
  → 写入 wrapper.grad_gate/up/down_proj_buf
  → KTMoEFunction.backward 把这三个 tensor 返回给 Autograd
  → AccumulateGrad 为 Parameter.grad 建立/累积另一份梯度
  → GAS>1 时再执行大范围 aten::add_
  → optimizer 消费 Parameter.grad
```

由于 wrapper 一直持有 C++ gradient buffer，Autograd 不能安全地把这个临时梯度当作唯一所有者直接“偷走”；PR 的 profiler 结论是旧路径形成了第二份 base gradient，并在累积时执行大 `aten::add_`。

### 4.2 新路径

```text
C++ 预分配 BF16 gradient buffer
  → 注册为 Parameter 对应的 authoritative grad_view
  → 第一个 microbatch：C++ overwrite，成功后 Parameter.grad = grad_view
  → 后续 microbatch：C++ 在同一 buffer 原地 accumulate
  → KTMoEFunction 对 base Parameter 返回 None，不再进入 AccumulateGrad
  → optimizer 直接读取该 C++ buffer
  → optimizer.step 后 Parameter.grad = None，只关闭窗口，不清 buffer
```

这里的“authoritative”表示 optimizer 看见的 `.grad` 与 C++ 写入的是同一块 storage，不再存在“C++ 临时梯度 + PyTorch optimizer 梯度”两份大对象。

### 4.3 optimizer window / GAS 状态机

| 时刻 | Python 状态 | C++ 动作 | 目的 |
|---|---|---|---|
| 初始化后 | 所有 KT `Parameter.grad is None`，窗口 closed | 输出状态未初始化 | 禁止 optimizer 读取未写 buffer |
| 窗口第一个 microbatch | closed | 完整初始化或按 expert 覆盖/延迟清零 | 生成本窗口第一份有效梯度 |
| C++ backward 成功 | `.grad` 全部指向注册的 `grad_view`，窗口 open | 提交状态 | optimizer 可见 |
| GAS 后续 microbatch | open | `old + scale × current` 原地累积 | 代替 PyTorch `aten::add_` |
| `optimizer.step()` 后 | `.grad=None` | 不清内存，只关闭窗口 | 下一窗口进入 overwrite/lazy-clear |
| pointer 变化或前次失败 | closed | 下一次强制完整 clear/overwrite | 不读取未知或部分写入的数据 |

Python 同时固定并检查 dtype、device、shape、stride、storage offset 和 data pointer；发现 `.grad` 被外部 tensor 替换、部分参数为 `None` 而部分仍 alias、buffer 被 resize 或异步 submission 重入时会 fail fast，而不是静默训练。

### 4.4 为什么 lazy clear 不会正常读取错误 buffer

新实现不是直接删除 `memset`，其正确性依赖以下覆盖关系：

1. **首次使用、输出指针变化、失败恢复：** 全量 clear，未初始化的 `torch.empty` 内容不会参与计算。
2. **新 optimizer window 的 active base expert：** dWeight 对完整 Gate/Up/Down expert slice 覆盖写，不读取旧值。
3. **上一窗口 active、当前第一个 microbatch inactive 的 expert：** 单独清零，防止 stale gradient 被 optimizer 读取。
4. **本窗口后续 microbatch：** 只有在 Python `.grad` 仍完整 alias、C++ state 已成功提交且 pointer 未变化时才允许 accumulate；此时旧值就是本窗口先前 microbatch 的合法梯度。
5. **LoRA：** LoRA kernel 有 read-modify-write，第一 microbatch 会清上一窗口 active expert 的 LoRA slice，再开始本窗口累积，不能套用 base dWeight 的纯覆盖假设。
6. **异常：** C++ 在 dispatch 前先把状态标成 invalid，只有 NUMA 计算和 TP merge 全部成功后才 commit；失败后的下一次调用回到全量初始化路径。

因此，在这些不变量成立时不会因“没有全量 clear”读到错误梯度。真正的风险是外部训练框架破坏生命周期，例如替换部分 `.grad`、绕过 step 后的 `update_kt_lora_pointers()`、改变 storage/pointer，或者混用 authoritative 与 legacy wrapper；代码选择报错而不是猜测。

还要注意，`TpBwdBufferClear` 并未消失：每次 backward 仍要清 per-TP sparse FP32 partial、`grad_input`/router partial 等临时 buffer。被消除的是稳定 optimizer window 中约 54 GiB 持久 base-gradient 的全量扫描。

### 4.5 Full、LoRA、Hybrid 与分布式

- Full base weight、fused expert LoRA 和 PEFT LoRA 都能注册 authoritative grad view；Hybrid 同时管理 base 与 LoRA。
- 只对 `AMXBF16_SFT && num_gpu_experts==0` 开启；quantized backend、GPU expert 和 SkipLoRA 保持 legacy lifecycle。
- 分布式时各 rank 的 routed rows/`grad_output` gather 到 rank 0；rank 0 C++ 生成唯一 optimizer gradient，并只对 parameter gradient 乘 `1/world_size`。
- `dX` 和 selected router-weight gradient scatter 回各 rank，继续各自 GPU Autograd；普通 GPU `lora_experts` 仍由 DDP/FSDP 管理。
- authoritative 路径不再对 KT CPU gradient 做第二次 all-reduce，避免重复规约和 collective hang；非 rank 0 不应持有 KT optimizer gradient。

## 5. `e26e21f`：`torch.zeros` 改为 `torch.empty`

提交：[e26e21f](https://github.com/kvcache-ai/ktransformers/commit/e26e21f0a263006161755c268b45f808428968ef)

三个 Full-FT gradient buffer 从：

```python
torch.zeros(..., dtype=torch.bfloat16, device="cpu")
```

改为：

```python
torch.empty(..., dtype=torch.bfloat16, device="cpu")
```

Qwen3-30B-A3B 的精确规模是：

```text
128 experts × 768 I × 2048 H × 3 projections × 48 layers × 2 bytes
= 57,982,058,496 bytes
= 54.0 GiB
```

`empty` 避免 Python 分配阶段先对这 54 GiB 做一次 eager zero/first-touch；C++ 在第一次 backward 仍执行强制完整初始化，所以这不是“首次清零也取消了”。其收益是删除一次重复的 Python 侧内存扫描，并让 C++ worker 按自己的并行/NUMA 路径完成必要 first-touch。legacy backend 也仍走 C++ 完整 clear 分支，不会直接使用 `empty` 的随机内容。

## 6. CPU/GPU 内存影响

以下是 Qwen3-30B-A3B、48 层、seq=4096、top-k=8、2 NUMA、默认 cache depth=2 的静态估算；不是 RSS 实测。

| 项目 | 本地 `1e95053` | PR #2094 | 变化 |
|---|---:|---:|---:|
| C++ authoritative/base grad buffer | 54.0 GiB | 54.0 GiB | 0；两边都需要真正的 expert gradient |
| PyTorch `Parameter.grad` 第二份 base gradient | 约 54.0 GiB | 0 GiB，直接 alias C++ buffer | 约 -54.0 GiB |
| checkpoint C++ cache | 0.844 GiB shared | 40.50 GiB per-layer | +39.656 GiB |
| cached CPU expert output | 0 | 0.750 GiB | +0.750 GiB |
| 上述主要项净变化 | — | — | **约 -13.594 GiB host RAM** |

这张表解释了两个看似矛盾的事实：checkpoint reuse 单独会明显增加 host cache，但 authoritative gradient 又删除更大的一份 PyTorch gradient，因此组合后的 Full-FT host 峰值预计略降，而不是增加约 40 GiB。

边界条件：

- `Parameter.grad` 是否形成完整第二份物理页以运行时 Autograd/allocator 为准；约 54 GiB 是按旧路径和 PR profiler 结论得到的容量估算，应通过 RSS、storage pointer 与 profiler 实测确认。
- `kt_max_cache_depth=1` 若通过验证，PR cache 可从 40.50 GiB 降为 20.25 GiB，组合净节省会进一步扩大。
- checkpoint reuse 仅 pure Full 单进程开启；LoRA/Hybrid/分布式没有这 40 GiB cache 增量。
- GPU gradient checkpoint 没关闭，因此预计不会增加 11–13 GiB 的完整 no-checkpoint GPU activation 集合；GPU 侧主要是原有 recompute transient 和 cached CPU output 的回传。
- `torch.empty` 主要改变初始物理页触达时机，不改变最终需要的 54 GiB C++ buffer 虚拟容量。

## 7. 对 backward、3× 目标和 TPS 的实际意义

### 7.1 明确会减少的工作

- checkpoint recompute 中的 CPU input staging、CPU MoE forward kernel 和对应等待；
- dWeight 对 input/route-weighted `dY` 的重复 panel packing；
- 旧 Autograd 为 Full base gradient 建立第二份 `.grad` 的大复制；
- GAS>1 时 PyTorch 对三组 base gradient 的大 `aten::add_`；
- 稳定 optimizer window 中对完整 persistent base/LoRA gradient 的 zero scan；
- 初始化阶段 Python `torch.zeros` 的 54 GiB eager first-touch。

### 7.2 不会被该 PR 消除的工作

- GPU attention、norm 和 router 的 checkpoint recompute；
- cached CPU expert output 的 CPU→GPU 传输；
- CPU MoE backward 的 Down dX、activation、Gate/Up dX、router grad 和 base dWeight 本体；
- 每次 backward 的 temporary partial-buffer clear；
- optimizer 的参数更新；
- optimizer 后从更新的 row-major BF16 Parameter 重新生成 AMX forward/backward BufferB 的 reload/repack。

### 7.3 不能直接套用旧打点做减法

历史 `20260720_130854...` 的 `checkpoint_recompute=1.991 s` 是整个 decoder-layer 重算边界，不是 CPU MoE forward 的独占时间。#2086 只跳过其中 CPU MoE 部分，因此不能直接从 6.671 s backward 中减掉完整 1.991 s。

同理，历史 `buffer_clear=0.336 s` 来自旧实现和旧 recorder；新实现仍清 temporary buffers，且 first step/pointer recovery 仍会全量初始化，不能预先把 0.336 s 全部归零。authoritative `AccumulateGrad/add_` 的收益还可能位于旧 C++ wrapper timing 之外，旧拆解没有单独量化。

此外，首次 forward 现在承担 cache/output 保存流量，部分耗时会从 backward 移到 forward。若只看 `backward / forward`，可能出现比 step/TPS 更乐观的改善。因此 PR #2094 是否足以把 backward 压到 forward 的 3 倍以内，当前源码分析仍不能给出确定答案；必须在相同 commit/build/config 下重新测：

```text
forward
backward
完整 step / TPS
checkpoint cached CPU MoE
optimizer-grad overwrite / accumulate / lazy-clear bytes
AccumulateGrad / aten::add_
base reload / backward repack
CPU RSS 与 GPU max_memory_allocated/reserved
```

PR 描述没有提供 `1e95053` 与 `e26e21f` 的同配置端到端 TPS A/B 数值，只提供正确性、smoke 和 profiler 现象，因此不能把“方向合理”写成“已证明达到 3×”。

### 9.3 自动评审意见的归类

当前只有自动 reviewer 的 `COMMENTED`，没有 approval。其三项意见中：

1. `thread_local` dWeight timing 被“主线程读取”的担忧：实际 `InNumaPool::process_tasks()` 在每个 worker 上调用 `finalize_func_(thread_id)`，finalizer 读取的是各 worker 自己的 thread-local counter；按当前 WorkerPool 实现，评论所假设的“只在主线程 finalize”不成立，但后续重构仍应保留这一约束。
2. `from_bb_transposed` 的 `K_STEP/N_STEP` stride 可读性问题位于本地 `1e95053` 已存在的代码，不是这 3 个提交的新差异；当前二者同为 32，不改变结果，但最好用语义正确的常量降低未来 tile shape 分化风险。
3. legacy 分布式 `grad_buf.copy_(grad_gpu.cpu())` 的临时 CPU tensor 优化建议只作用旧同步分支；#2094 的 authoritative rank-0 路径在此前已经 return，不经过该 copy。

### 9.4 仍需重点验证的集成风险

- patched Trainer 必须在每个真实 `optimizer.step()` 后调用 `update_kt_lora_pointers()`，使 `.grad` alias 释放并把 base/LoRA weight 标记 dirty；其他训练框架接入时不能漏掉该 hook。
- gradient clipping、AMP/GradScaler、ZeRO/FSDP、`zero_grad(set_to_none=...)` 或用户 callback 不得把一部分 KT `.grad` 替换成新 tensor；完整设为 `None` 或保持原 alias 才是合法状态。
- checkpoint reuse 默认会把 CPU cache 从 shared 改为 per-layer，必须把 host RSS 和 NUMA placement 纳入验收；不能只观察 GPU 显存。
- GAS>1 应特别覆盖“后续 microbatch 首次激活新 expert”和“跨 optimizer window expert 集合变化”，验证 lazy clear 的 active-expert union。
- 分布式应验证 parameter gradient 的 `1/world_size` 只应用一次，router/GPU 参数仍由既有 DDP/FSDP 语义处理。
- BF16 原地累积会在每个 microbatch store 时发生 BF16 rounding；PR 报告已对 GAS 1/2/4 做 FP32 reference，但长 GAS/长期收敛仍应单独观察。

## 11. 最终判断

PR #2086/#2094 的设计正面命中了当前 Full-FT backward 的三个内存流量问题：checkpoint 时重复的 CPU expert forward、Autograd 的第二份 54 GiB base gradient/大 `add_`、以及每个 optimizer window 的全量 gradient clear。状态机对首次使用、inactive expert、GAS、pointer 变化和异常恢复都有显式保护，因此它不是以牺牲已知正确性换取“直接不清 buffer”。

代价主要是 pure Full 单进程 checkpoint 模式默认保留 48 层 C++ cache，增加约 40.406 GiB host 内存；与删除约 54 GiB PyTorch gradient duplicate 抵消后，主要项净值预计仍减少约 13.594 GiB。GPU checkpoint 仍在，预计不会出现完整关闭 checkpoint 的大幅显存增长。

从源码上看值得采用并做实测，但当前不能断言它已把 backward 压到 forward 的 3 倍以内，也不能给出确定 TPS 增益：PR 没有提供同配置端到端 A/B 数字，GitHub KT-Kernel CI 又在环境准备阶段失败。最可靠的下一步是完整同步 3 个线性提交、重建，并用新 profiler 对相同 Qwen3 配置重新建立性能基线。
